```text
NemotronHMoE

  → MoeRunnerBackend.FLASHINFER_TRTLLM
```

# 一、根据启动参数推导实际分支

你的命令没有设置：

```bash
--moe-a2a-backend
--moe-runner-backend
--expert-parallel-size
```

因此预计为：

```text
moe_a2a_backend = none
moe_runner_backend = auto
ep_size = 1
tp_size = 4
```

## 1. `get_moe_impl_class()` 返回 `FusedMoE`

你代码中：

```python
self.experts = get_moe_impl_class(quant_config)(...)
```

在没有启用 DeepEP、Mooncake、MORI、NIXL 等 A2A backend 时，`get_moe_impl_class()` 返回普通的：

```python
FusedMoE
```

因此不会走 `DeepEPMoE`。你的 routed experts 是 TP=4 的 FusedMoE，而不是 EP dispatch 路径。

## 2. quant method 是 `ModelOptNvFp4FusedMoEMethod`

该模型的 MoE 权重是 ModelOpt NVFP4，因此 `FusedMoE` 初始化时会得到：

```python
self.quant_method = ModelOptNvFp4FusedMoEMethod(...)
```

SGLang 的 ModelOpt 配置在遇到 `FusedMoE + NVFP4` 时明确返回这个 quant method。

## 3. `auto` 默认解析成 `FLASHINFER_TRTLLM`

在 SGLang `v0.5.12.post1` 中：

```python
if moe_runner_backend.is_auto():
    moe_runner_backend = MoeRunnerBackend.FLASHINFER_TRTLLM
```

也就是说，没有显式传 `--moe-runner-backend` 时，NVFP4 MoE 默认选择 FlashInfer TRT-LLM。

当前 SGLang `main` 仍然采用同样选择：Blackwell NVFP4 MoE 的 `auto` 默认解析成 `FLASHINFER_TRTLLM`。

# 二、完整调用路径

由于你关闭了 CUDA Graph，正常调用路径是：

```text
NemotronHMoE._forward_core_normal
  │
  ├─ router_logits = torch.mm(...)
  ├─ topk_output = self.topk(...)
  ├─ hidden_states = fc1_latent_proj(hidden_states)
  │
  └─ self.experts(hidden_states, topk_output)
         │
         ▼
      FusedMoE.forward
         │
         ▼
      FusedMoE.forward_impl
         │
         ├─ StandardDispatcher.dispatch
         │
         ├─ run_moe_core
         │     └─ quant_method.apply
         │
         └─ StandardDispatcher.combine
```

`FusedMoE.forward_impl()` 明确执行 `dispatcher.dispatch → run_moe_core → dispatcher.combine`，而 `run_moe_core()` 直接调用 `self.quant_method.apply()`。

随后进入：

```text
ModelOptNvFp4FusedMoEMethod.apply
  → self.runner.run(...)
  → registered fused function:
       ("none", "flashinfer_trtllm")
  → fused_experts_none_to_flashinfer_trtllm_fp4
  → trtllm_fp4_block_scale_moe
```

该注册函数根据 `quant_info` 的类型确认走 FP4 分支。

# 三、为什么你没有抓到 `topk_ids`

这是目前非常重要的一个发现。

当 backend 是：

```text
flashinfer_trtllm
```

SGLang 的 `TopK.forward_cuda()` 不执行普通的 `select_experts()`，而是直接返回：

```python
BypassedTopKOutput(
    hidden_states=hidden_states,
    router_logits=router_logits,
    topk_config=self.topk_config,
)
```

也就是说，SGLang 此时没有产生：

```text
topk_ids
topk_weights
```

路由计算被推迟到 FlashInfer TRT-LLM kernel 内完成。

而你的抓数代码是：

```python
if hasattr(topk_output, "topk_weights") and hasattr(topk_output, "topk_ids"):
    ...
```

`BypassedTopKOutput` 只有：

```text
hidden_states
router_logits
topk_config
```

没有 `topk_weights/topk_ids`，所以这一段自然不会执行。

这很可能就是你目前只看到：

```text
moe_router_logits
moe_experts_output
```

却看不到 `moe_topk_ids/weights` 的原因。
